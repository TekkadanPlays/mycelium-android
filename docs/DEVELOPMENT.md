# Mycelium Android — Development Guide

> Last verified against source: v0.5.03 (versionCode 41), Kotlin 2.2.0, AGP 8.13.0

## Prerequisites

| Requirement | Version | Notes |
|-------------|---------|-------|
| JDK | 11+ | `compileOptions` targets Java 11 |
| Android SDK | API 35+ | `minSdk = 35`, `compileSdk = 36`, `targetSdk = 36` |
| Android Studio | Ladybug+ | Or command-line build tools with Gradle 8.13 AGP support |
| Git | 2.x | Submodules used for Cybin library |

> **Note:** `minSdk = 35` means Mycelium runs only on Android 15+. This is intentional — the app targets modern devices and avoids legacy API compatibility overhead.

## Clone & Setup

```bash
# Clone with submodules (Cybin is an includeBuild, not a git submodule currently,
# but other directories like lightning-kmp, phoenix, etc. are submodules)
git clone --recursive https://github.com/TekkadanPlays/mycelium-android.git
cd mycelium-android
```

If you already cloned without `--recursive`:

```bash
git submodule update --init --recursive
```

### Keystore (Release Builds)

Release signing is configured via `keystore.properties` at the project root. If this file is missing, release builds fall back to the debug keystore automatically.

To set up release signing, create `keystore.properties`:

```properties
storeFile=/path/to/your/keystore.jks
storePassword=your_store_password
keyAlias=your_key_alias
keyPassword=your_key_password
```

This file is **not** checked into version control.

### Local Properties

`local.properties` is generated by Android Studio and points to your SDK location. It's gitignored. If building from the command line without Studio, create it:

```properties
sdk.dir=/path/to/Android/Sdk
```

## Project Structure

The project is a multi-module Gradle build with two primary modules:

```
mycelium-android/
├── app/                    # Android application module
├── cybin/                  # Nostr protocol library (composite build)
├── benchmark/              # Baseline profile module (currently disabled)
├── docs/                   # Documentation
├── gradle/                 # Gradle wrapper + version catalog
│   └── libs.versions.toml  # Central dependency version catalog
├── build.gradle.kts        # Root build file (plugin declarations only)
├── settings.gradle.kts     # Module inclusion + Cybin composite build
└── gradle.properties       # JVM args, parallel build, caching flags
```

### Module Relationship

`settings.gradle.kts` includes Cybin as a **composite build** with dependency substitution:

```kotlin
includeBuild("cybin") {
    dependencySubstitution {
        substitute(module("com.github.TekkadanPlays:cybin")).using(project(":cybin"))
    }
}
```

This means `app/build.gradle.kts` declares `implementation("com.github.TekkadanPlays:cybin:0.1.0")` but Gradle resolves it to the local `cybin/` source. Changes to Cybin are picked up immediately — no publish step needed.

## Building

### Debug Build

```bash
./gradlew assembleDebug
```

Output: `app/build/outputs/apk/debug/app-debug.apk`

Debug builds use:
- Application ID suffix: `.debug` (so `social.mycelium.android.debug`)
- Version name suffix: `-debug`
- ProGuard/R8: disabled
- `WALLET_DEV_MODE = true`

### Release Build

```bash
./gradlew assembleRelease
```

Output: `app/build/outputs/apk/release/app-release.apk`

Release builds use:
- Full R8 minification and resource shrinking
- ProGuard rules in `app/proguard-rules.pro` (extensive keep rules for Compose, Ktor, serialization, Material Icons, secp256k1)
- `WALLET_DEV_MODE = false`
- Signing from `keystore.properties` or fallback to debug keystore

### Install

```bash
adb install app/build/outputs/apk/release/app-release.apk
```

### Compose Compiler Metrics

Composition metrics are generated automatically with release builds:

```bash
./gradlew assembleRelease
# Output at: app/build/compose_metrics/
```

This produces `app_release-composables.csv`, `app_release-classes.txt`, and `app_release-module.json` for analyzing recomposition stability and skippability.

```properties

org.gradle.parallel=true
org.gradle.caching=true
org.gradle.configureondemand=true

android.nonTransitiveRClass=true
```

### Kotlin Compiler Options

Configured in `app/build.gradle.kts`:

```kotlin
kotlinOptions {
    jvmTarget = "11"
    freeCompilerArgs += listOf(
        "-opt-in=androidx.compose.material3.ExperimentalMaterial3Api",
        "-opt-in=androidx.compose.foundation.ExperimentalFoundationApi",
        "-opt-in=androidx.compose.foundation.layout.ExperimentalLayoutApi",
    )
}
```

### Compose Compiler

```kotlin
composeCompiler {
    enableStrongSkippingMode = true
    reportsDestination = layout.buildDirectory.dir("compose_metrics")
    metricsDestination = layout.buildDirectory.dir("compose_metrics")
}
```

Strong skipping mode is enabled — composables with unstable parameters can still be skipped if their values haven't changed (equals check).

## Room Database

The app uses Room for persistent caching. Database is defined in `db/AppDatabase.kt`.

**Entities:**
- `CachedProfileEntity` — kind-0 profile metadata
- `CachedNip65Entity` — kind-10002 relay lists
- `CachedNip11Entity` — NIP-11 relay information documents
- `CachedEventEntity` — Generic event cache
- `CachedFollowListEntity` — kind-3 follow lists
- `CachedEmojiPackEntity` — Custom emoji pack definitions

**DAOs:** `ProfileDao`, `Nip65Dao`, `Nip11Dao`, `EventDao`, `FollowListDao`, `EmojiPackDao`

Room is used as a **cache layer** — the primary data flow is still WebSocket → memory. Room provides persistence across process death and faster cold starts.

## Background Service

`RelayForegroundService` maintains persistent WebSocket connections when the app is backgrounded. It's declared as a `specialUse` foreground service type in the manifest.

Three connection modes are available (configured in `NotificationPreferences`):

| Mode | Behavior |
|------|----------|
| **Always On** | Foreground service runs with persistent notification. WebSockets stay open. |
| **Adaptive** (default) | No foreground service. WorkManager (`RelayCheckWorker`) periodically checks inbox relays for new events. |
| **When Active** | Connections only exist while the app is in the foreground. Zero background activity. |

## Startup Orchestration

`StartupOrchestrator` coordinates phased startup so relay subscriptions don't all fire at once:

| Phase | Priority | What Loads |
|-------|----------|------------|
| 0 — Settings | CRITICAL | Kind-30078 synced settings (NIP-78) |
| 1 — User State | HIGH | Kind-3 follow list + kind-10000 mute list |
| 2 — Feed | HIGH | Main kind-1 subscription + self NIP-65 |
| 3 — Enrichment | NORMAL | Outbox feed, notifications, profiles, counts, bookmarks, emoji packs |
| 4 — Background | LOW | DMs (kind-1059), background notification subs |

Each phase gates on the previous one completing (or timing out).

## Common Workflows

### Adding a New Screen

1. Create the composable in `ui/screens/YourScreen.kt`
2. Add a `composable("your_route")` block in `MyceliumNavigation.kt`
3. If it needs a ViewModel, create it in `viewmodel/` and scope it appropriately (see Architecture doc for scoping rules)
4. Wire navigation from existing screens via `navController.navigate("your_route")`

### Adding a New Repository

1. Create the class in `repository/YourRepository.kt`
2. Most repositories are singletons (`object`) or have a singleton `getInstance()` pattern
3. If the repository needs relay subscriptions, use `RelayConnectionStateMachine.getInstance().requestTemporarySubscription()` or `requestOneShotSubscription()`
4. If it should be per-account, integrate with `AccountScopedRegistry`

### Adding a New Nostr Event Kind

1. Add the kind constant and handler registration in `RelayConnectionStateMachine` (kind routing)
2. Create or update the relevant repository to process the events
3. If it needs a subscription, decide the priority level and whether it's temporary, one-shot, or part of the main feed
4. Update the `StartupOrchestrator` phase if needed

### Working with Cybin

Cybin changes are picked up immediately by the app build (composite build). To work on Cybin:

1. Edit files in `cybin/cybin/src/main/java/com/example/cybin/`
2. Run `./gradlew assembleDebug` from the root — Gradle builds both modules
3. No publish or version bump needed for local development

## Troubleshooting

### JVM Heap Errors

If you see `java.lang.OutOfMemoryError` during builds, increase the Gradle daemon heap in `gradle.properties`:

```properties
org.gradle.jvmargs=-Xmx6144m -Dfile.encoding=UTF-8
```

### Benchmark Module

The `benchmark/` module is commented out in `settings.gradle.kts` due to an AGP version conflict with the baseline profile plugin. To re-enable it, update AGP or use a compatible `androidx.baselineprofile` plugin version.

### hs_err_pid*.log Files

JVM crash logs in the project root (`hs_err_pid*.log`, `replay_pid*.log`) are from Gradle daemon crashes during development. They're safe to delete and are gitignored.

### Debug vs Release Behavior

- Debug builds have `WALLET_DEV_MODE = true` — the embedded Lightning wallet connects to testnet
- Release builds have `WALLET_DEV_MODE = false` — mainnet
- Debug builds skip ProGuard, so reflection-based code (Ktor, serialization) works without keep rules
- If something works in debug but crashes in release, check `app/proguard-rules.pro`
